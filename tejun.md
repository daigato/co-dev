# GitHub共同開発手順書

このドキュメントでは、安全かつスムーズに共同開発を進めるためのGit/GitHubの手順を解説します。

## 💡 共同開発の3大鉄則
1. **`main` ブランチには直接コミット・プッシュしない！**
   - 必ず作業用の「ブランチ」を作って開発します。これにより、相手の作業中のコードを壊すリスクをなくします。
2. **作業を始める前は必ず `git pull` で最新にする！**
   - 他の人が進めた作業を取り込んでから自分の作業を始めないと、後で衝突（コンフリクト）しやすくなります。
3. **こまめにコミット・プッシュする！**
   - 小さい単位で保存しておくことで、バグが起きたときに戻しやすくなり、相手も進捗を把握しやすくなります。

## 🔄 日常の開発フロー

機能追加やバグ修正を行うときの、一連の流れです。

### 1. 準備：最新のコードを取り込む
作業を始める前に、自分のパソコン（ローカル）にあるコードを最新状態にします。
```bash
# 1. mainブランチに切り替える
git checkout main

# 2. GitHubから最新のコードをダウンロードする
git pull origin main
```

### 2. 作業用ブランチを作る
新しく作る機能や修正内容に合わせて、作業用の「ブランチ（枝分かれ）」を作ります。
- ブランチ名は、何をするかわかりやすい名前にします（例: `feature/login-ui`, `fix/broken-link`）。
```bash
# 新しいブランチを作って、自動的にそのブランチに切り替える
git checkout -b ブランチ名
```

### 3. コードを書いてコミットする
エディタでコードを書きます。キリの良いところでコミット（履歴の保存）をします。
```bash
# 1. 変更したファイルをステージング（登録）する
git add .

# 2. コミットする（メッセージは「何をしたか」を日本語で分かりやすく）
git commit -m "〇〇機能のUIを作成"
```
※ 作業が終わるまで、この「コードを書く ➔ `git add` ➔ `git commit`」を何回か繰り返します。

### 4. GitHubにプッシュする
作業が完了したら、作成したブランチをGitHub（リモート）にアップロードします。
```bash
# 自分のブランチをGitHubに送る
git push origin ブランチ名
```

### 5. プルリクエスト (PR) を作成する
GitHubのWebサイトに行き、プッシュしたブランチから `main` ブランチへの **Pull Request (PR)** を作成します。
- タイトルや説明に、何を作ったか、動作確認した内容を書きます。
- **もう1人のメンバーにレビュー（確認）を依頼します。**

### 6. レビューとマージ
- **レビューする人**: 送られてきたPR of コードを確認し、実際に自分のPCで動かしてみるなどして、問題なければ「Approve (承認)」します。
- **PRを作った人**: 承認されたら、GitHub上で「Merge pull request」ボタンを押して、`main` ブランチに合流させます。
- 合流が終わったら、GitHub上の不要になったブランチは削除してOKです。

### 7. ローカル（自分のPC）を片付ける
マージが終わったら、自分のパソコンも最新状態に戻し、使い終わったブランチを削除して整理します。
```bash
# 1. mainブランチに戻る
git checkout main

# 2. 合流された最新のコードを取り込む
git pull origin main

# 3. 使い終わったローカルブランチを削除する
git branch -d ブランチ名
```

## ⚡ コンフリクト（衝突）が起きてしまったら？

同じファイルの同じ行を2人が同時に編集してマージしようとすると、Gitが「どちらを優先すればいいかわからない！」とエラーを出します。これが **コンフリクト（衝突）** です。

### 解決の手順（VS Codeを使う場合）
1. コンフリクトが発生したファイルを開くと、VS Code上で以下のような表示になり、どちらを採用するか選べるようになります。
   - `Accept Current Change`（現在の変更を受け入れる：自分が書いたコードを残す）
   - `Accept Incoming Change`（入力側の変更を受け入れる：相手が書いたコードを残す）
   - `Accept Both Changes`（両方の変更を取り込む）
2. 2人で相談して、どちらのコードを残すか（あるいは両方組み合わせるか）を決め、VS Codeのボタンをクリックするか、直接ファイルを編集します。
3. 修正が終わったら、競合が解決したことをGitに伝えてコミット・プッシュします。
   ```bash
   git add 解決したファイル名
   git commit -m "conflictを解消"
   git push origin ブランチ名
   ```

## 🛠️ 初心者におすすめのツール・設定
コマンドライン（黒い画面）での操作が不安な場合は、以下のツールを使うと視覚的にGitを操作できておすすめです。

1. **VS CodeのGit機能 (拡張機能: GitLens など)**
   - ソース管理タブから、ボタンクリックで `add` や `commit`、`push/pull` ができます。コンフリクトの解決も視覚的に行えます。
2. **GitHub Desktop**
   - 公式のデスクトップアプリです。ブランチの作成やプッシュ、プルリクエスト作成がGUI（画面操作）だけで完結します。
